# Proposal: Add Step-By-Step Onboard Command

## Summary

Add a root `getsuperpower onboard` command that guides a user through first-run
workspace setup one step at a time.

The command should ask whether the user wants to set up:

- RTK, so Codex shell commands can use token-reduced command output.
- CodeGraph, so the current codebase can be indexed for structural code
  questions.

The v1 flow should be explicit and local-first: show what will be checked, ask a
yes/no question for each setup area, run only the confirmed step, and print the
next command or result before moving on.

## Motivation

GetSuperpower depends on a good local agent environment. Today that setup lives
in repository instructions and prior operator knowledge, so a new user has to
discover the pieces manually. An `onboard` command can make the first-run path
feel guided:

- RTK can reduce shell-output token usage during Codex work.
- CodeGraph can make structural codebase questions cheaper and more accurate.
- The CLI can turn those setup choices into a repeatable checklist.
- Users stay in control because each step is confirmed before it runs.

## Scope

In scope:

- Add a root `getsuperpower onboard` command.
- Keep the command in the root-first GetSuperpower surface, not under the
  removed nested `getsuperpower` subcommand.
- Ask whether the user wants RTK setup.
- Ask whether the user wants CodeGraph setup.
- Check whether `rtk` appears available before suggesting or running RTK setup.
- Check whether `.codegraph/` exists before suggesting or running CodeGraph
  indexing.
- Present setup as a step-by-step flow with clear skipped, already-ready, and
  completed states.
- Support non-interactive test seams so the prompt flow and command-runner
  behavior can be tested without touching the real home directory or indexing
  the real repo.
- Add focused tests for command registration, prompt branching, skip behavior,
  already-ready behavior, and confirmed CodeGraph indexing.
- Run the standard repo gate before delivery.

Out of scope:

- Reintroducing paused Pony Trail history, revert, prehook, or Ponyrace command
  surfaces.
- Installing arbitrary global packages without a user confirmation.
- Depending on files generated by a smoke test.
- Adding a new npm dependency.
- Requiring RTK or CodeGraph for normal `install`, `clone`, `deps`, `validate`,
  or `list` commands.

## Proposed Design Direction

Add `configureOnboardCommand` in `src/getsuperpower.ts` and register it from the
existing root command configuration path. Keep `src/cli.ts` as the thin shell
that wires real prompts and command execution into the command module.

The command should model onboarding as a small list of setup steps:

1. RTK token-reduction setup.
2. CodeGraph indexing setup.

Each step should:

- print what it is checking;
- detect whether the tool/state is already available;
- ask the user whether to continue when setup is still needed;
- run only the confirmed setup action;
- print the exact follow-up command when the action cannot be completed
  automatically.

For v1, CodeGraph can be executable because the project instruction already
names the command shape: `codegraph init -i`. RTK installation may be
environment-specific, so the command should first check `rtk --version`; if RTK
is missing, it should ask the user whether to see/install the recommended RTK
step and then use the configured command runner or print exact instructions,
depending on the final approved design.

## Acceptance Criteria

- `getsuperpower onboard` appears in root command registration and help.
- The command asks the user whether RTK setup is needed when RTK is not already
  available.
- The command reports RTK as already ready when `rtk --version` succeeds.
- The command asks the user whether CodeGraph indexing is needed when
  `.codegraph/` is absent.
- The command reports CodeGraph as already ready when `.codegraph/` exists.
- When the user declines a step, the command prints that the step was skipped
  and continues to the next step.
- When the user confirms CodeGraph setup, the command runs `codegraph init -i`
  in the target project directory through an injectable runner.
- Confirmed setup failures produce clear messages that identify the failed step
  and preserve the underlying command failure detail.
- Tests do not invoke the real `rtk`, mutate the real home directory, or build a
  real CodeGraph index.
- Existing GetSuperpower commands continue to register and behave as before.
- `rtk bun run check` passes before delivery.

## Open Questions For Review

- Should v1 actually execute RTK installation when RTK is missing, or should it
  print the exact RTK setup command because RTK installation is environment
  specific?
- Should `onboard` default to the current working directory only, or accept
  `--dir <dir>` like `install`, `list`, and other project-local commands?
- Should there be a `--yes` option for trusted local automation, or should v1
  stay fully interactive?
